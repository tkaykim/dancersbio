댄서들의 프로필, 포트폴리오를 등록 및 수정 할 수 있도록 하는 기능을 하면서 (안무 제작 이력, 공연 이력, 수상 이력, 방송 출연 이력, 대회 이력, 심사 이력, 수업 이력 등 댄서 또는 인플루언서로의 할 수 있는 일들에 대한 본인의 경력과 사진, 대표영상 등을 보여줄 수 있는 포트폴리오 페이지)



해당 프로필을 보고 클라이언트가 댄서들에게 일을 제안할 수 있고, (안무제작, 심사 섭외, 출연 섭외 등등)

댄서들은 클라이언트에게 받은 일들을 가지고 동의 여부를 확인할 수 있으며, 동의한 프로젝트에 대해 파트너 댄서(함께 할 댄서들 예를 들면 안무가가 댄서들을 모집) 하는 등의 클라이언트, 안무가, 댄서 들 간의 업

구상하고 계신 시스템은 단순한 구인·구직 사이트를 넘어, 댄스 씬의 특수성을 반영한 **'댄서 매니지먼트 및 에이전시 워크플로우 자동화 플랫폼'**으로 보입니다. 그리고 엔터테인먼트의 효율적인 운영 기조에 맞춰, 클라이언트-안무가-댄서 사이의 복잡한 커뮤니케이션 리소스를 최소화하는 방향으로 구조를 제안해 드립니다.

이 시스템을 크게 세 가지 핵심 모듈로 나누어 정리해 보았습니다.

1. 댄서 포트폴리오 & 경력 관리 모듈
댄서의 정체성을 시각적으로 보여줌과 동시에, 클라이언트가 검색하기 용이하도록 데이터를 규격화하는 것이 핵심입니다.

카테고리화된 이력 관리: 안무, 공연, 수상, 방송, 심사, 수업 등 항목별로 경력을 입력하여 클라이언트가 필터링(예: "방송 경험이 있는 안무가")하기 좋게 구성합니다.

미디어 통합: YouTube/Vimeo 링크 연동을 통한 대표 영상 노출 및 Supabase Storage를 활용한 고화질 프로필 사진 관리 기능을 제공합니다.

동적 업데이트: 댄서가 직접 본인의 최신 이력을 수정할 수 있는 대시보드를 제공하여 데이터의 선도를 유지합니다.

2. 클라이언트 섭외 및 제안 관리 모듈 (Casting Workflow)
수많은 연락 수단(DM, 카카오톡, 이메일)으로 파편화된 섭외 과정을 시스템 안으로 통합합니다.

다이렉트 제안서 송신: 클라이언트가 특정 댄서의 프로필에서 즉시 '업무 제안' 버튼을 눌러 예산, 일정, 장소, 요구사항을 담은 제안서를 보낼 수 있습니다.

상태 추적 시스템:

제안됨(Pending): 댄서에게 알림톡/메일 발송.

검토 중(Reviewing): 댄서가 내용을 확인한 상태.

동의/거절(Accepted/Declined): 댄서의 최종 의사 결정.

표준 계약 연동: 동의 시 자동으로 기본적인 업무 협약서나 가이드라인이 발송되도록 설계하여 법적 리스크를 줄입니다.

3. 프로젝트 협업 및 팀 빌딩 모듈 (Partner Recruiting)
안무가가 메인 디렉터가 되어 팀을 구성해야 하는 경우를 위한 기능입니다.

파트너 모집 공고: 제안을 수락한 안무가가 해당 프로젝트 내에서 함께 할 댄서(파트너)를 모집하는 내부 공고를 올립니다.

인벤토리 기반 매칭: 시스템 내 등록된 댄서들에게 프로젝트 참여 가능 여부를 묻는 알림이 전송되며, 안무가는 신청자들의 프로필을 보고 원클릭으로 팀을 확정합니다.

통합 일정 관리: 확정된 팀원들에게 연습 일정, 촬영 장소 등의 정보를 공유하고 구글 캘린더 등과 연동하여 노쇼(No-show)를 방지합니다.

🛠 기술적 구현 제안 (대표님의 스택 기반)
대표님께서 선호하시는 효율적인 로직과 자동화 구조를 적용한다면 다음과 같은 구성이 가능합니다.

구분	적용 기술	역할
Frontend	Next.js / React	댄서들의 포트폴리오 페이지 및 클라이언트 예약 UI
Backend	Supabase	댄서 데이터베이스, 인증, 실시간 매칭 상태 관리
Automation	n8n	섭외 제안 시 알림톡 발송, 일정 확정 시 구글 캘린더 등록 자동화
Communication	Slack / Alimtalk	댄서와 클라이언트 간의 상태 변화 실시간 알림
이 시스템이 구축되면 **"불필요한 커뮤니케이션을 줄이고 업무 효율화에 집착"**하시는 대표님의 업무 스타일에 부합하는 강력한 비즈니스 툴이 될 것입니다.

혹시 이 시스템의 **첫 번째 단계로 구현하고 싶은 구체적인 데이터 구조(예: DB 테이블 설계)**나, 클라이언트가 보게 될 제안서 양식에 대해 먼저 상세히 기획해 드릴까요?

프론트엔드만을 놓고 보세요

당신은 댄스업계를 완벽하게 이해하고 있고, 세계 최고의 실리콘밸리 출신 프론트 엔드 개발자이자, 프론트엔드 유저 경험 베이스 디자인에 특화된 전문가입니다.

아래 조건들을 만족하는 유저플로우와 페이지 구성들을 기획하세요



1. 댄서들의 입장에서 본인의 프로필(사진, 영상, 경력 등)을 등록하거나 수정할 수 있어야 합니다.

2. 관리자가 이미 누군가 댄서의 프로필을 등록해둔 경우도 있음. 이 경우 댄서가 본인인증을 한 후 해당 건에 대해 수정권한을 획득하고 수정할 수 있어야 합니다

3. 받은 제안들을 확인할 수 있는 섹션이 있어야합니다. 제안 리스트, 제안 상세 등을 확인할 수 있어야 합니다.

4. 클라이언트는 소속 회사, 담당자 이름 등을 입력한 상태로 제안해야합니다.

5. 댄서는 매니저를 지정할 수 있습니다. (해당 매니저가 프로필 관리, 제안 관리를 할 수 있습니다)

6. 매니저는 여러명의 댄서들을 관리할 수 있습니다.

실리콘밸리의 설계 철학은 **"복잡한 비즈니스 로직을 사용자에게는 가장 단순한 직관으로 제공하는 것"**에 있습니다. 댄서의 예술적 감성과 매니저/클라이언트의 비즈니스 효율성을 동시에 만족시키는 최상위 레벨의 프론트엔드 유저 플로우와 페이지 구성을 제안합니다.

1. Core User Flow: Identity & Permission
가장 복잡한 권한 체계(댄서-매니저-관리자)를 매끄럽게 연결하는 핵심 흐름입니다.

[A] 프로필 소유권 확보 플로우 (Claiming Flow)
검색 및 발견: 댄서가 플랫폼 접속 후 관리자가 미리 등록한 본인의 프로필 발견.

소유권 요청: '본인인가요? 수정 권한 요청하기' 버튼 클릭.

본인 인증: SNS(Instagram API) 연동 또는 신분증 업로드.

승인 대기: 관리자 대시보드에서 승인 시, 해당 프로필은 댄서의 계정과 '귀속(Mapped)' 처리.

온보딩: 기존 데이터 검토 및 즉시 수정 모드 진입.

[B] 매니지먼트 위임 플로우 (Delegation Flow)
매니저 초대: 댄서가 이메일/ID로 매니저 초대.

권한 설정: 프로필 수정권한, 제안 확인권한, 수락/거절 권한 등 세부 토글 설정.

매니저 수락: 매니저는 본인의 대시보드에서 '관리 중인 댄서' 리스트에 해당 댄서 추가 확인.

2. Page Architecture: The "Spotlight & Dashboard"
Page 1. [Public] 댄서 포트폴리오 (The Spotlight)
클라이언트에게 보여지는 최종 결과물입니다. 편집 모드에서는 실시간으로 수정 가능한 Canvas UI를 채택합니다.

Hero Section: 고화질 프로필 이미지 및 대표 퍼포먼스 영상(Auto-play loop).

Career Timeline: [안무/공연/방송/수상/심사] 카테고리별 탭 전환형 이력 리스트.

Media Grid: YouTube/Instagram 피드 연동 및 갤러리.

Booking Widget (Fixed): 클라이언트가 언제든 제안을 시작할 수 있는 플로팅 버튼.

Page 2. [Private] 댄서/매니저 워크스페이스 (The Hub)
댄서와 매니저가 공유하는 비즈니스 관리 공간입니다.

Multi-Profile Switcher: (매니저용) 관리 중인 여러 댄서의 프로필을 상단 드롭다운으로 즉시 전환.

Proposal Board (Kanban/List View): * New: 클라이언트가 방금 보낸 제안 (Red Dot 알림).

In Review: 매니저/댄서가 확인 중인 건.

Accepted/Declined: 최종 의사결정 완료 건.

Quick Editor: 이력 한 줄 추가, 사진 순서 변경 등 드래그 앤 드롭 방식의 편집기.

Page 3. [Client] 섭외 제안 페이지 (The Bridge)
클라이언트의 신원을 명확히 하고 댄서가 의사결정하기 편한 고품질 데이터를 수집합니다.

Client Identity Form: 소속(회사/브랜드명), 담당자 성함/직함, 연락처 필수 입력.

Project Brief: 프로젝트 성격(안무제작/출연 등), 예상 예산, 일정(D-Day), 장소.

Requirement Tags: 필요한 댄서 수, 스타일, 의상 컨셉 등 상세 옵션 체크박스.

3. UI/UX Specialist's Point (디테일 설계)
기능	기획 의도 및 UX 전략
Inline Editing	별도의 '수정 페이지'로 이동하지 않고, 포트폴리오 페이지에서 즉시 텍스트를 클릭해 수정하는 방식(Notion 스타일).
Proposal Summary Card	제안 리스트에서 예산, 날짜, 장소를 요약 카드로 보여주어 댄서가 한눈에 우선순위를 판단하게 함.
Permission Badge	현재 보고 있는 페이지가 '매니저 권한'인지 '본인 권한'인지 상단 바에 명확히 표기하여 실수 방지.
Verification Shield	본인 인증이 완료된 댄서 프로필 옆에 'Verified' 배지를 부여하여 신뢰도 상승.
4. 프론트엔드 기술 스택 및 확장성 고려
실리콘밸리 표준에 맞춘 성능 최적화 전략입니다.

State Management: 매니저가 여러 명의 댄서를 전환할 때 데이터 로딩 지연이 없도록 React Query 또는 SWR을 통한 캐싱 처리.

Role-Based Access Control (RBAC): 프론트엔드 단에서 UserRole에 따른 UI 렌더링을 엄격히 구분 (ManagerView vs DancerView).

Responsive Design: 댄서들은 주로 이동 중에 모바일로 제안을 확인하므로, Mobile-First 대응 필수.

이 기획안은 대표님께서 지향하시는 **"실수를 줄이고 반복을 없애는 자동화 시스템"**의 프론트엔드 청사진입니다.

다음 단계로, 매니저가 여러 명의 댄서를 한 화면에서 관리할 때 가장 효율적인 '통합 일정/제안 대시보드'의 와이어프레임을 상세히 그려드릴까요? 혹은 클라이언트 제안서 입력 단계의 유효성 검사 로직을 먼저 살펴볼까요?

관리자가 등록해놓은 프로필 목록에 없다면 본인이 신규 등록도 가능해야합니다.

새로운 요구사항을 포함하여, 관리자가 등록한 기존 데이터를 활용하는 케이스와 완전히 새로운 댄서가 진입하는 케이스를 모두 포괄하는 '통합 온보딩 및 워크스페이스' 유저 플로우를 설계했습니다.

실리콘밸리 스타일의 UX 핵심은 **"탐색(Discovery)에서 소유(Ownership)까지의 마찰을 제로(Zero)로 만드는 것"**입니다.

1. Unified Onboarding: "Find or Create" Flow
사용자가 처음 진입했을 때 본인의 존재 여부를 확인하고, 권한을 획득하거나 새롭게 생성하는 관문입니다.

Step 1: Identity Search (검색 및 확인)

입력창에 활동명(Stage Name)을 입력합니다.

Case A (데이터 있음): 관리자가 미리 등록한 프로필 카드들이 나타납니다. "이게 저예요!" 버튼을 눌러 본인 인증(Claim) 절차로 진입합니다.

Case B (데이터 없음): "검색 결과가 없나요? 당신의 첫 번째 공식 포트폴리오를 만드세요."라는 문구와 함께 [신규 등록] 버튼이 활성화됩니다.

Step 2: Verification vs. Registration

Claim 시: 인스타그램 연동 혹은 활동 증빙 자료 업로드 후 관리자 승인을 기다립니다. (승인 전에도 '임시 수정'은 가능하게 하여 리텐션을 유지합니다.)

신규 등록 시: 기본 인적 사항부터 시작하는 **프로그레시브 위저드(Progressive Wizard)**가 시작됩니다.

2. Page Configuration: The Professional Suite
[Page 1] Profile Creator (신규 및 수정 모듈)
단순한 폼 입력이 아닌, '빌더(Builder)' 형식의 인터페이스를 제공합니다.

Blocks System: 경력, 영상, 사진 등을 '블록' 단위로 추가/삭제/드래그 앤 드롭 이동 가능.

Media Importer: YouTube URL만 넣으면 썸네일과 제목을 자동으로 긁어오는 파싱 기능.

Live Preview: 수정 중인 내용이 실제 공개 페이지에서 어떻게 보일지 우측에서 실시간 확인.

[Page 2] Inbox: Proposal Management (제안 관리 섹션)
댄서와 매니저가 비즈니스에만 집중할 수 있는 대시보드입니다.

Proposal Card List: * 헤더 정보: 클라이언트 기업 로고, 회사명, 담당자 이름, 프로젝트 제목.

Status Tags: 검토 필요, 매니저 확인 중, 승인 완료, 정산 대기 등.

Proposal Detail View:

클라이언트가 입력한 상세 기획안(Brief) 확인.

Action Buttons: [수락], [거절], [메시지 보내기(문의)].

Manager Memo: 매니저와 댄서만 볼 수 있는 내부용 비공개 메모장.

1. Public Profile Page (The Viral Hook)

Copy: "댄서님이신가요? 3분 만에 당신의 커리어 포트폴리오를 완성하세요."

2. Onboarding: "Search & Claim" Flow

Step 1: The Search Gate

UI: 거대한 검색창 하나만 존재. "활동명(Dancer Name)을 입력하세요."

Logic:

Case A (이미 등록됨): 검색 결과 카드에 Claim Profile 버튼 활성화. -> [본인 인증/권한 요청] 프로세스로 이동.

Case B (없음): "등록된 댄서가 없네요. 새로운 별을 등록할까요?" -> [신규 생성] 프로세스로 이동.

Step 2: Verification (Claiming Process)

Identity Proof: 인스타그램 공식 계정으로 로그인(OAuth) 또는 신분증/명함 업로드.

Trigger: 제이의 화려한 이력을 보고 감탄하던 중, 하단에 "지수님도 댄서인가요? 멋진 포트폴리오를 만들어보세요" 배너를 발견합니다.

Delegation: 일이 바빠진 지수는 아는 지인(매니저)을 초대해 [매니저 권한 부여].

Proposal: 며칠 뒤, 클라이언트가 지수의 페이지에서 [제안하기] 클릭 -> 매니저에게 알림톡 도착 -> 매니저가 내용 확인 후 수락.
